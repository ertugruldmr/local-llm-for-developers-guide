Hardware · what can you actually run?

Open-weight LLMs sized against the hardware needed to serve them — from a 1B edge model to a ~1 trillion-parameter MoE that needs a GPU node. Pick a precision and a GPU; the bars and the orange line redraw to show what fits in one unit and what needs many — or flip View → GPU units to see the raw card count as a row of GPU icons. Closed frontier models map to a rough capability tier (they don't publish sizes), shown dashed and labeled "est."

The gpt-oss trick: sized to the GPU

OpenAI ships gpt-oss-120b with its MoE expert weights in MXFP4 (~4.25 bits/param). Those experts are 90%+ of the weights, so the whole 117B model lands near ~60–65 GB — under one 80 GB H100/MI300X. gpt-oss-20b was sized the same way to fit a 16 GB consumer card. The architecture was tuned to the hardware unit, not the other way around.

80 GB is the packing unit

Datacenter serving rounds up to whole 80 GB (H100) or 141 GB (H200) GPUs. DeepSeek-V3 at native FP8 is ~670 GB of weights → it overflows an 8×H100 node (640 GB) but fits an 8×H200 node (1,128 GB). Flip precision to Q4 and it drops to ~4–6 GPUs.

MoE saves compute, not memory

A Mixture-of-Experts model only computes its active params per token (Kimi K2.6: 32B of ~1T; DeepSeek-V4-Pro: 49B of 1.6T), so it's fast for its size — but every expert must still be resident in VRAM. Capacity is set by total params; throughput by active params.

Apple: capacity vs bandwidth

A 512 GB M3 Ultra (~384 GB usable) can hold a model that otherwise needs a multi-GPU server — on one quiet desktop. But its ~819 GB/s is ~¼ of an H100's bandwidth, so tokens/sec are far lower. Great for one developer; not for high-concurrency serving.

Open weight → closed capability tier

Rough mapping of where each open tier lands relative to the closed frontier. This is a capability ballpark, not a benchmark or a size claim — closed models don't publish parameter counts, and rankings shift monthly. Verify on your own task before trusting it.

How the numbers are computed

Serve estimate = total params × bytes-per-param × 1.2 (the 20% covers KV cache + activations + fragmentation at modest context). Bytes/param: BF16 = 2.0, FP8/INT8 = 1.0, Q4 ≈ 0.5, MXFP4 ≈ 0.55. Native uses the precision each model actually ships in (gpt-oss → MXFP4, DeepSeek/Kimi/Mistral-Large-3 → FP8, the rest → BF16) — flip to Q4 to see the typical local footprint people run.

GPU count = ⌈serve ÷ unit capacity⌉. Apple unit capacity is ~70% of nominal unified memory (OS reserves the rest). Long context (>32K) and high concurrency grow KV cache sharply — treat these as weight-fit floors, add headroom for production. MoE keeps all experts resident, so memory tracks total params even though only active params compute.
